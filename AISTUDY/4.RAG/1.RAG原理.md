# RAG 原理

## 什么是 RAG

RAG（Retrieval-Augmented Generation）即检索增强生成，是一种将外部知识引入大语言模型的技术。

### 核心思想

- **检索**：从知识库中检索与用户问题相关的文档片段
- **增强**：将检索到的文档片段作为上下文提供给大语言模型
- **生成**：大语言模型基于上下文生成回答

## RAG 的工作流程

```
用户提问 → 文本向量化 → 在向量数据库中搜索相似文档 → 把文档 + 问题一起发给 AI → AI 基于文档回答
```

### 详细步骤

1. **用户提问**：用户输入自然语言问题
2. **文本向量化**：将问题转换为向量表示（Embedding）
3. **相似度搜索**：在向量数据库中搜索与问题向量最相似的文档片段
4. **上下文构建**：将检索到的文档片段与原问题组合成新的提示词
5. **AI 生成**：大语言模型基于上下文生成回答
6. **返回结果**：将回答返回给用户

## RAG 的优势

- **知识更新**：不需要重新训练模型，只需更新知识库
- **减少幻觉**：基于真实文档回答，减少模型编造信息的可能
- **领域专业**：可以针对特定领域的知识进行定制
- **成本效益**：比微调大模型更经济实惠

## RAG 的应用场景

- **企业知识库问答**：内部文档、手册、政策问答
- **智能客服**：基于产品文档和FAQ回答用户问题
- **文档助手**：PDF、Word、Markdown等文档的智能问答
- **学术研究**：论文、专利、文献的查询和总结
- **法律助手**：法律法规、案例的检索和解释

## RAG 的关键技术挑战

- **文档分块**：如何将长文档分割成合适的片段
- **向量质量**：Embedding 模型的选择和优化
- **检索策略**：如何提高检索的准确性和相关性
- **上下文管理**：如何在有限的上下文窗口中包含最相关的信息
- **评估方法**：如何评估 RAG 系统的性能

## RAG 与微调的区别

| 特性   | RAG         | 微调         |
| ---- | ----------- | ---------- |
| 知识更新 | 简单（更新知识库）   | 复杂（重新训练）   |
| 领域适配 | 快速（添加领域文档）  | 较慢（需要标注数据） |
| 成本   | 低（主要是API调用） | 高（训练和推理）   |
| 适用场景 | 知识密集型任务     | 特定任务的性能优化  |

## 常见的 RAG 框架

- **LlamaIndex**：专注于 RAG 的框架，提供完整的文档处理 pipeline
- **LangChain**：通用 LLM 应用框架，包含 RAG 相关组件
- **Haystack**：由 deepset 开发的 NLP 框架，支持 RAG
- **FastGPT**：开源的知识库问答系统，基于 RAG 技术

## LlamaIndex RAG Demo 技术分析

### 1. 运用到的技术

- **LlamaIndex框架**：用于简化RAG系统的构建，提供文档加载、向量化、索引构建和查询功能
- **通义千问大模型**：
  - 语言模型：`qwen-max`，用于生成回答
  - 嵌入模型：`text-embedding-v1`，用于文本向量化
- **向量数据库**：LlamaIndex内置的向量存储（基于Chroma）
- **文档处理**：使用`SimpleDirectoryReader`加载和处理文档
- **Python编程**：使用Python实现整个RAG流程

### 2. 处理过的数据存储位置

- **临时文档文件**：程序首先在当前目录创建了8个`sample_doc_*.txt`文件，存储示例文本
- **向量存储**：处理后的数据（向量）存储在LlamaIndex内置的向量数据库中，默认是：
  - **内存中**：在开发和测试环境中，Chroma默认使用内存存储
  - **本地磁盘**：如果指定了持久化路径，会存储到本地磁盘

### 3. 使用的索引技术

- **向量索引**：LlamaIndex使用Chroma的向量索引，具体是：
  - **HNSW（Hierarchical Navigable Small World）**：这是Chroma默认使用的索引算法
  - **优势**：查询速度快、内存占用合理、支持高维向量
  - **工作原理**：通过构建层次化的图结构，实现近似最近邻搜索

### 4. 向量相似度计算

- **余弦相似度**：LlamaIndex默认使用余弦相似度计算向量间的相似度
- **计算方式**：
  - 公式：`cosine_similarity = dot_product(vector1, vector2) / (norm(vector1) * norm(vector2))`
  - 范围：\[-1, 1]，值越大表示相似度越高
- **优势**：对向量长度不敏感，适合文本等语义相似度计算

### 5. 向量维度选择

- **由嵌入模型决定**：向量维度由使用的嵌入模型决定
- **通义千问text-embedding-v1**：
  - 维度：通常为1536维（具体以模型实际输出为准）
  - **选择依据**：
    - 足够捕捉文本语义信息
    - 平衡计算效率和存储成本
    - 与语言模型的输入要求匹配
- **影响因素**：
  - 维度越高，语义表达能力越强，但计算和存储成本也越高
  - 维度越低，计算速度越快，但可能损失一些语义信息

### 技术流程总结

1. **文档创建**：生成8个示例文档文件
2. **文档加载**：使用`SimpleDirectoryReader`加载文档
3. **文本向量化**：使用通义千问的`text-embedding-v1`模型将文本转换为向量
4. **索引构建**：使用Chroma的HNSW索引算法构建向量索引
5. **查询处理**：
   - 将问题向量化
   - 使用余弦相似度在向量数据库中搜索相似文档
   - 将检索到的文档作为上下文提供给语言模型
   - 语言模型基于上下文生成回答

通过这些技术的组合，Demo实现了一个完整的RAG系统，能够基于外部知识回答用户问题。
